# AIOps-Sanitizer: 基于多Agent协作的分布式系统“隐式故障”长链推理与自动化自愈系统

AIOps-Sanitizer 是一个专为微服务和分布式系统（如 Kubernetes 集群）设计的智能自动化运维系统。项目核心放弃了传统的“单 Prompt 吞吐”模式，构建了一个包含**长链推理**与**多 Agent 协作**的自动化闭环自愈系统。

## 核心痛点解决
1. **“隐式故障”难以捕捉**：许多由于配置漂移、内存渐进式泄漏、死锁或偶发性竞态条件（Race Condition）引起的性能劣化，在早期以“低噪日志”形式存在，人工排查如大海捞针。
2. **根因分析（RCA）链路极长**：一个前端请求的延迟，可能经过了网关、鉴权、多级缓存、中间件到数据库，排查需要横跨多台机器的上下文日志，传统的单 Agent 往往因为上下文窗口限制或缺乏深度推理能力，给出似是而非的结论。
3. **“知行脱节”矛盾突出**：现有的 AI 运维助手大多停留在“给出排查建议”的阶段，由于缺乏严格的闭环验证机制，工程师不敢直接让 AI 去修改线上配置或代码，导致故障响应时间（MTTR）依然长达数小时。

---

## 核心逻辑流与多Agent架构设计

系统由四个核心 Agent 组成，采用类似生产消费者与黑板模式（Blackboard Pattern）进行高强度协同：

### 1. 观测与上下文汇聚 Agent (Observability Agent)
该 Agent 持续挂载在集群的后端守护进程中，利用 RAG（检索增强生成）技术与精简的向量数据库，对海量日志流进行实时语义过滤。当捕捉到异常的长尾延迟或特定异常调用栈时，它会自动收集当前时间戳前后 5 分钟内，横跨网络、系统、应用三层的全部拓扑数据和上下文快照，打包成“故障现场包”。

### 2. 根因深度推理 Agent (Reasoning Agent - 核心长链推理)
针对复杂的分布式故障，该 Agent 引入了思维树（Tree of Thoughts, ToT）与 ReAct（Reasoning and Acting）框架：
* 它不会直接给出结论，而是先提出 3-4 个可能的故障假说（例如：假说 A 为 Redis 连接池耗尽，假说 B 为宿主机 I/O 瓶颈）。
* 随后，它通过长链推理，像经验丰富的架构师一样，主动调用预设的内部 API 工具（如：查询当前连接数、执行 iostat、调取网卡丢包率）。
* 经过多轮“推理-行动-观察-修正”的深度思考链路，最终排除干扰项，锁定引发故障的底层根因（例如：“由于 Upstream 服务的 Keep-Alive 超时配置与 Nginx 错配，导致高并发下 TIME_WAIT 状态套接字堆积，最终引发连接拒绝”）。

### 3. 决策与修补 Agent (Patching Agent)
接收到根因报告后，该 Agent 负责生成具体的修复动作。为了保证安全，它不直接操作生产环境，而是通过静态分析和 AST（抽象语法树）解析，生成对应的配置修复 PR（如优化 Terraform/Kubernetes YAML 配置文件）或代码修复补丁，并附带详细的修改合规性解释。

### 4. 灰度闭环验证 Agent (Verification Agent)
这是系统最关键的防线。它会自动将修补方案部署到 Staging（预发）环境或 K8s 的 Shadow Pod（影子 Pod）中，自动合成并注入故障发生时的流量负载，运行端到端的集成测试。同时，它会持续监控该隔离环境的指标。只有当测试 100% 通过且各项业务指标恢复正常时，才会最终合并代码并向运维团队发送“故障已自动治愈”的通知；若验证失败，则自动回滚并触发降级报警。

## 落地成果与实际业务价值

该项目目前已在团队内部以及开源社区的 3 个中型微服务集群中灰度落地：

* **Token 消耗与处理深度**：由于引入了深度长链推理和多轮工具交互，单次复杂故障的排查与自愈过程需要消耗约 **80万 至 150万 Token**。系统每日在后台进行主动巡检、长链推理与策略仿真，**日均稳定消耗 800万 至 1200万 Token**。这证明了长链推理在处理复杂系统架构时的必要性与高强度吞吐能力。
* **时效性断崖式提升**：成功将复杂分布式故障的平均排查与响应时间（MTTR）从过去的 **45 分钟断崖式下跌至 2.5 分钟**。90% 的隐式配置错配和资源瓶颈可以在用户毫无感知的情况下完成自动闭环。
* **自动化闭环高准确率**：在近一个月的运行中，系统成功自主拦截并修复了 14 起偶发性的中间件连接雪崩故障。在长链推理的加持下，**根因定位的准确率达到了 92.4%**，自动生成的修复补丁在验证阶段的**通过率达到了 85% 以上**。
